# 酒店竞对监控 v0.2 第一阶段设计

## 目标

在不增加任何 OTA 网页抓取、不引入境外服务、不消耗测试额度的前提下，修复当前链路中会导致错误日报、错误口径或错误推送状态的高优先级问题，使正式 Automation 具备可靠的失败停止能力和可审计的数据来源说明。

## 方案比较

### 方案 A：只修运行状态

只检查子进程退出码，避免采集失败后继续生成日报。改动最小，但房型未落实、酒店错配、脱敏价误算、前端旧配置等问题仍会产生看似成功但口径错误的日报。

### 方案 B：可靠性与数据真实性一起修复

同时处理失败传播、当次产物校验、床型参数、酒店身份匹配、脱敏价、百度额度账本、前端下载和推送联动。改动范围可控，并直接覆盖本次审查发现的 P0/P1 问题。

### 方案 C：一次性重写全部采集架构

把逐家 FlyAI 查询改成商圈批量搜索，新增稳定观察组和周期性候选刷新。长期收益最大，但需要验证批量返回的酒店匹配质量，不适合与可靠性热修同时上线。

本阶段选择方案 B。方案 C 作为 v0.2 第二阶段，在第一阶段稳定后单独设计和实测。

## 范围

### 1. 运行状态必须真实

- `run-once.ps1` 调用 `install.ps1` 和 `run-api-mvp.ps1` 后必须检查 `$LASTEXITCODE`。
- 子进程失败时，外层必须返回非零退出码，并写入 `Status: failed`，不能写 `Status: ok`。
- 每次启动采集前记录开始时间；只有本次运行后新生成的报告输入才能被写入 `LatestReportInput`。
- 正式运行成功后，报告必须包含本次 `RunId`、`GeneratedAt`、`DryRun: False` 和实际产物路径。
- `verify-local.ps1` 同样检查子进程退出码；被跳过的步骤写 `skipped`，不能写 `ok`。

### 2. 查询口径必须区分“已应用”和“仅记录”

- 把 `query.roomType` 映射为 FlyAI 的 `--hotel-bed-types`：大床类映射 `king`，双床类映射 `twin`，无法识别时不传并标记 `unsupported`。
- 当前 FlyAI CLI 不支持房间数、成人数和儿童数参数。这些字段继续作为经营口径记录，但报告必须明确标记为 `not-applied-by-flyai-cli`，模型不得声称价格已按人数筛选。
- 报告新增 `AppliedQueryScope`，列出日期、床型和未应用字段。
- 历史查询指纹继续包含全部用户口径，避免不同经营口径误比较；日报分析只能把实际应用的字段描述为价格查询条件。

### 3. 酒店身份必须校验

- 高德本店文本检索不能直接取第一条。优先精确名称，其次使用标准化名称包含关系；没有兼容结果时停止正式采集。
- FlyAI 结果不能直接取 `itemList[0]`。先按标准化酒店名匹配；没有兼容结果时将该候选标记为 `identity-mismatch`，不写入价格。
- 报告输出 FlyAI 实际匹配到的酒店名和身份匹配状态，便于审计。
- 本阶段不做模糊编辑距离和跨城市猜测，避免错误匹配看起来更智能。

### 4. 脱敏价格不能作为精确数字

- `¥3xx`、`¥1xx` 等价格的精确数值返回空值或 0，不参与最大价过滤、价格比例、竞品分层和涨跌计算。
- 脱敏价格只保留原始字符串和价格带说明。
- 历史比较中，任一侧为脱敏价时输出“价格带不可精确比较”，不能计算涨跌额和百分比。

### 5. 百度额度必须按自然日持久化

- 在 `data/usage/baidu-YYYY-MM-DD.json` 保存当日真实 HTTP 调用数。
- 每次调用前读取当日账本，调用后原子写回；同一天多次运行共享同一上限。
- DryRun 不创建或修改正式额度账本。
- 报告同时输出“本次使用”和“今日累计使用”，避免把单次计数误写成每日计数。

### 6. 前端配置必须使用当前表单值

- 下载配置时直接读取当前表单，不复用初始化缓存。
- 表单字段变化后自动刷新预览，或在下载前强制重新生成。
- `pushMode=clawbot`、`wecom`、`none` 必须生成不同的 Automation 推送指令。
- `competitorCount`、`maxCandidates`、半径、晚数、人数和额度字段在生成前执行范围校验；无效值时阻止下载并显示具体字段。
- 将“品牌补漏关键词”改成“品牌加权关键词”，直到第二阶段真正实现补充搜索。

### 7. 推送能力表述必须可验证

- WorkBuddy Automation 官方可确认的通知路径写为“推送到 WorkBuddy 小程序”。
- ClawBot 写为“绑定后可接收助理任务结果；Automation 是否可直接选择 ClawBot 以当前桌面端实际界面为准”。
- `pushMode=clawbot` 时提示用户在首次试跑中确认微信实际收到消息；未确认前不能写“推送成功”。
- 企业微信群 webhook 保留为代码层可验证的备用推送。

## 不在本阶段处理

- 不把 20 次逐家 FlyAI 查询改成批量查询。
- 不新增稳定观察组、每周候选刷新或人工锁定竞对。
- 不抓点评正文，不恢复携程或其他 OTA 浏览器自动化。
- 不自动修改 WorkBuddy GUI、Automation 或 ClawBot 绑定状态。
- 不修改用户私有 `config/hotel-monitor.json`，只扩展示例配置和迁移检查。

## 测试策略

- 先新增故障注入测试，证明子采集器失败时 `run-once.ps1` 当前错误返回成功。
- 新增 DryRun 行为测试，验证床型映射、未应用人数口径和酒店身份匹配状态。
- 新增脱敏价格回归测试，确保脱敏价不进入数值比较。
- 新增百度跨两次运行共享每日额度的本地测试，全部使用本地夹具，不调用百度。
- 新增前端源代码与本地浏览器交互测试，验证修改字段后预览和下载数据一致，三种推送模式生成不同提示词。
- 完整运行 `npm test`、PowerShell 语法解析、`verify-local.ps1` 和更广的敏感信息扫描。

## 验收标准

1. 故障注入时外层退出码非零，运行报告为 `failed`，且不会指向旧的 latest 输入。
2. 大床房查询的 FlyAI 命令包含 `--hotel-bed-types king`；双床房包含 `twin`。
3. 报告明确说明房间数、成人数和儿童数未被当前 FlyAI CLI 应用。
4. FlyAI 返回其他酒店时不写入候选价格。
5. `¥3xx` 不会被解析为 `3`，也不会产生精确涨跌。
6. 同一天两次运行共享百度额度账本，累计调用不超过配置上限。
7. 前端修改城市后无需额外点击即可生成当前城市配置；选择 `none` 后提示词不要求 ClawBot 推送。
8. 所有验证均为零额度，本阶段不调用高德、FlyAI 或百度正式接口。

## 上线顺序

1. 合并第一阶段代码并运行零额度验收。
2. 更新 GitHub 主仓库。
3. 在 WorkBuddy 实际 Automation 工作空间执行 `git pull`。
4. 备份并迁移私有配置，但不覆盖用户酒店与查询口径。
5. 手动正式试跑一次，确认报告输入、日报和实际推送状态。
6. 再恢复每天 07:30 的无人值守 Automation。
